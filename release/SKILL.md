# Release

You cut a release for the project. Most projects already have release automation — your job is to find it, understand it, and drive the release through it. Only do things manually that the existing tooling doesn't cover.

## Step 1: Understand the project's release process

Before doing anything, figure out what automation already exists. This is the most important step — **do not skip it**.

Check for:
- **GitHub Actions / CI workflows** — look in `.github/workflows/` for release, publish, or deploy workflows. Read them to understand what they do: version bumping, changelog generation, asset building, GitHub Release creation, deployment. Many projects have workflows that handle most or all of the release process when triggered.
- **Release scripts** — check `package.json` scripts, `Makefile`, `composer.json` scripts, or similar for release-related commands.
- **Versioning scheme** — check existing tags with `git tag --sort=-version:refname` to identify the pattern (semver, calver, etc.).
- **Changelog convention** — is there a `CHANGELOG.md`? Is it auto-generated by CI or manually maintained?
- **Release trigger** — what kicks off the release? A tag push? A workflow dispatch? A PR merge to a release branch? A manual `gh release create`?

Also check for a **`.release.md`** file in the repo root. This is an optional project-specific config that provides context this skill can't infer: target audience, changelog category names, announcement format preferences, post-release steps, or any other project-specific release guidance. If it exists, follow it.

**Tell the user what you found.** Describe the existing release pipeline before proposing any actions. Your role in Step 5 depends entirely on what the automation handles vs what it doesn't.

## Step 2: Determine what's being released

Find everything that changed since the last release:

```bash
git log --oneline <last-tag>..HEAD
```

Categorize the changes:
- **Features** — new capabilities (`feat:` commits)
- **Fixes** — bug fixes (`fix:` commits)
- **Breaking changes** — anything that changes existing behavior or public APIs
- **Other** — refactors, docs, chores, tests (may or may not belong in the changelog depending on project convention)

If there are no meaningful changes since the last release, say so and stop.

## Step 3: Determine the version

If the user provided a version in `$ARGUMENTS`, use it.

Otherwise, suggest a version based on the changes and the project's versioning scheme:
- **Semver:** breaking change → major, new feature → minor, fixes only → patch
- **Prerelease:** if the project supports prerelease types (dev, alpha, beta, rc), suggest the appropriate type and explain the progression
- **Other schemes:** follow the project's convention

Present the suggested version with reasoning. The user decides.

## Step 4: Craft the changelog

Whether the project auto-generates a changelog or not, the raw output (commit messages, PR titles) is rarely good enough to ship to users. Your job is to produce a polished, user-facing changelog.

Start from whatever is available — auto-generated changelog, commit log, PR list — and apply these transformations:

1. **Consolidate** — multiple commits that build up one feature or fix become one entry. Don't list the steps, describe the result.
2. **Rewrite for clarity** — remove jargon, PR references, commit hashes, and implementation details. Write in the language of the project's users, not its developers.
3. **Categorize** — prefix each item: **New:** for features, **Improvement:** for enhancements to existing behavior, **Fix:** for bug fixes. If `.release.md` defines different categories, use those instead.
4. **Order** — New → Improvement → Fix. Most impactful items first within each category.
5. **Drop internal-only changes** — refactors, test improvements, and CI changes don't belong unless they affect user-facing behavior.
6. **Call out breaking changes prominently** — these need migration guidance, separate from the regular categories.

Keep each item to one line. Keep the changelog proportional to the release — a small patch doesn't need a wall of text.

Present the draft to the user for review before proceeding. If the project's automation will generate its own changelog too (e.g. on the GitHub Release), the user may want to use your polished version to replace or supplement it.

## Step 5: Create the release

How you execute depends on what you found in Step 1.

### If the project has release automation (preferred path)

Your job is to **drive the automation**, not replace it. That means:

1. **Determine the right inputs** — what does the workflow need? A tag name? A version bump type? A release type? Map the version decision from Step 3 to the specific inputs the workflow expects.
2. **Show the user the plan** — "I'll trigger the release workflow with these inputs: [X]. The workflow will: [bump version, generate changelog, build assets, create GitHub Release]. Here's what that means for this release."
3. **Trigger it** — with the user's approval, trigger the workflow (via `gh workflow run`, tag push, or whatever the project uses).
4. **Follow up** — check the workflow run status. If it fails, help diagnose. If it succeeds, check the output (GitHub Release, assets, changelog) and flag anything that needs attention.
5. **Handle post-release steps** — some projects have separate deployment workflows, manual distribution steps, or announcements. Identify these in Step 1 and remind the user what comes next after the release is created.

If the automation produces a raw changelog, offer to replace it with the polished version from Step 4.

### If the project has no release automation (manual path)

Do it all:

1. **Update version references** — package.json, plugin headers, constants, wherever the version lives.
2. **Update the changelog file** — if the project maintains one
3. **Commit the version bump and changelog** — `chore: release vX.Y.Z`
4. **Tag the release** — `git tag vX.Y.Z` (match the project's tag format)
5. **Create a GitHub Release** — using `gh release create` with the changelog as the body

In either case: do not push tags, trigger workflows, or create releases without the user's explicit approval. Show them what will happen first.

## Step 6: Write the announcement

After the release is created, write a short announcement (2-4 sentences) summarizing the release for sharing — Slack, email, socials, community forums, wherever the project communicates with users.

Focus on the most impactful changes. Don't list everything — link to the full changelog for details. Match the tone and format to the project's audience (check `.release.md` for guidance).

If `.release.md` specifies announcement channels or formats, follow them.

## Rules

- Always check for existing release tooling first — don't replace automation that already works
- Always show the changelog draft before creating the release — changelogs are user-facing documents
- Never push tags or trigger deployments without explicit user approval
- Never include internal-only changes in a user-facing changelog
- If in doubt about the version number, ask — version numbers are promises to users
- If a `.release.md` exists, follow it — it's the project's release playbook
- Don't invent changes not in the commit log — the changelog is a record, not marketing
